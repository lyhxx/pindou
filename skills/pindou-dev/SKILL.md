# 保持项目定位

- 把项目视为纯前端拼豆图纸编辑器，不引入后端服务。
- 保持所有图片处理、画布编辑、量化转换、导出能力都在浏览器端完成。
- 优先服务“快速生成可开工的拼豆图纸，再进行手工修正”这个核心工作流。

# 遵守产品约束

- 保持首页为单屏入口，不扩展回长落地页。
- 保持编辑器为基于 viewport 的全屏工作台，页面本身不允许纵向滚动。
- 保持三栏结构：左侧输入与生成，中间舞台，右侧颜色与导出。
- 只允许画布缩放和平移发生在中间舞台内部，不能让画布把整页撑高或撑出滚动。
- 保持“上传图片调整”和“正式生成图纸”解耦。
- 只在用户点击“生成图纸”时覆盖当前图纸结果。
- 保持一个格子只对应一种拼豆颜色，不允许单格混色渲染。
- 避免增加只展示不工作的占位 UI。
- 保持文案为中文，避免中英混排和大段说明性废话。

# 理解当前信息架构

- 首页只承担工具入口作用，不负责完整产品介绍。
- 编辑器顶部只保留高频动作和项目信息，不重复放置侧栏已有能力。
- 左侧优先承接画布尺寸、图片上传、预览定位、生成参数。
- 中间区域是唯一的画布舞台和工具操作核心区。
- 右侧优先承接高级颜色、颜色替换、导入导出、颜色统计。

# 按既有工作流开发

## 画布与图片流程

- 先让用户设置或确认画布尺寸。
- 再让用户上传图片并在预览层调整缩放、平移、去背景等参数。
- 只在点击“生成图纸”后运行正式量化流程并更新 `BeadGrid`。
- 调整图片预览参数时，不要直接清空或覆盖当前图纸。

## 编辑与修图流程

- 画笔、橡皮、吸色、填充、框选都直接作用于网格数据。
- 颜色替换应优先基于已有颜色库 id，不要引入游离颜色值。
- 撤销和重做必须覆盖手工编辑、替换、整理画布等实际修改动作。

## 布局与舞台流程

- 修布局问题时，优先检查 `html`、`body`、`#root`、编辑器根节点的高度链条。
- 优先检查主工作台是否正确使用 flex 或 grid 分配剩余高度。
- 优先保证左右栏内部滚动，中间舞台 `overflow: hidden`。
- 不要把缩放 transform 施加到整页容器或三栏容器上。

# 优先处理这些已知方向

## 当前版本上下文

- 当前发布版本：`v1.0.3`
- 最近一轮调整重点在图片预览、帮助中心联动、图纸导出清晰度、裁切空白策略和顶部项目信息收敛
- 默认行为应保持为：左侧先看原图，只有用户主动开启实时预览或点击生成图纸时才做更重的图像处理
- 如果继续修改图纸生成逻辑，优先保证结果稳定，不要再默认叠加隐藏式边缘清理后处理

- 智能背景去除：从简单四角平均色升级为基于边缘种子的背景识别与泛洪去除。
- 边缘杂色一键清理：在量化后的 `BeadGrid` 上识别边缘跳色、小连通域和孤立噪点并自动替换。
- 仅处理边缘的颜色替换：让用户能只修边缘脏色，不误伤主体区域。
- 颜色清洗参数收敛：把去背景、容差、边缘清理等能力集中到图片模块。
- 撤销重做稳定性补强：确保历史栈不会被缓存恢复或生成流程意外清空。

# 修改时遵守这些实现原则

- 优先复用现有 store、现有类型和现有组件，不平行再造一套状态流。
- 优先在现有模块里补强，而不是新增同职责组件。
- 做图像处理时，先明确处理发生在“预览层”还是“正式生成层”。
- 做 UI 收缩时，优先删冗余入口、合并重复语义，而不是继续堆按钮。
- 做颜色相关功能时，始终围绕拼豆颜色库，而不是原图自由色值。
- 做导出能力时，优先保证图纸和工程文件准确，再考虑额外展示产物。
- 做帮助、提示、说明类改动时，统一维护在 `src/features/editor/helpContent.ts`，不要把长文案重新散写进页面组件。

# 验证要求

- 每次完成改动后运行 `npm run build`。
- 涉及编辑器布局、工具栏、舞台交互时，实际打开页面验证。
- 至少检查以下行为：
  - 页面本身是否还能滚动。
  - 画布放大后是否只在中间舞台内部变化。
  - 左右侧栏内容超出时是否是内部滚动。
  - 调整图片定位和缩放时是否误清空当前图纸。
  - 手工绘制后撤销和重做是否真实生效。
  - 颜色替换是否只替换目标颜色，不引入异常空格。
