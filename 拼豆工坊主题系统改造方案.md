# 拼豆工坊主题系统改造方案

## 1. 背景

当前项目已经有一层基础设计变量，主要集中在 [src/styles/tokens.css](/d:/code/pindou/src/styles/tokens.css)。  
但实际页面样式仍大量散落在 [src/styles/globals.css](/d:/code/pindou/src/styles/globals.css) 中，存在以下问题：

- 大量颜色、透明度、渐变、阴影直接写死
- 页面不同区域虽然都偏米白暖色，但语义不统一，视觉层级混乱
- 局部样式频繁改动后，整体气质越来越碎
- 后续如果想做“整体配色更换”或“用户可切换主题”，当前结构维护成本很高

这意味着：

- 现在不是简单“换一套颜色”就能解决问题
- 必须先补一层真正可维护的主题系统

## 2. 改造目标

本次主题系统改造的目标，不是先做很多主题，而是先做“能支持主题”的结构。

目标分为两层：

1. 主题架构目标
- 所有全局视觉基调都通过主题变量控制
- 页面主体、面板、弹窗、按钮、输入框、滚动条、顶部栏、舞台外壳等不再直接写死颜色
- 新增主题时不需要大面积改组件结构

2. 产品体验目标
- 默认主题先变得更统一、更像一个完整工具产品
- 后续可平滑扩展为 2 到 3 套主题
- 主题切换不影响拼豆颜色数据本身的可读性

## 3. 非目标

以下内容不属于这轮主题系统改造的核心范围：

- 不先做复杂“用户自定义配色器”
- 不先做多品牌运营后台
- 不把拼豆调色板颜色本身纳入页面主题切换
- 不在第一阶段同时重做所有布局结构

## 4. 当前样式问题归纳

### 4.1 token 与真实样式脱节

虽然 `tokens.css` 已定义：

- 基础文字色
- 页面/面板背景色
- 主次强调色
- 状态色
- 阴影
- 圆角

但 `globals.css` 中仍存在大量：

- `rgba(...)`
- `linear-gradient(...)`
- 直接十六进制颜色
- 只在局部使用、没有统一语义的背景与边框色

结果是：

- 同样是“面板”，不同模块用的不是同一套背景
- 同样是“选中态”，不同地方高亮规则不一致
- 同样是“浮层/卡片”，阴影和边框表达不统一

### 4.2 视觉语义没有分层

当前视觉变量缺少更细的语义拆分。例如：

- 页面背景
- 一级面板背景
- 二级面板背景
- 舞台背景
- 浮层背景
- 交互控件背景
- 弱/中/强边框

这些很多都还混在一起，导致：

- 某些区域太灰
- 某些区域太白
- 某些区域阴影过重
- 用户会觉得“整体丑”，而不是某个单独模块丑

### 4.3 页面主题与业务颜色耦合不清

项目里有两类颜色：

1. 界面主题颜色
- 页面背景
- 文字
- 边框
- 按钮
- 交互状态

2. 业务数据颜色
- 拼豆颜色库本身
- 图纸中的颜色结果
- 调色板统计结果

如果后续主题系统不明确隔离这两类颜色，就会出现：

- 主题一换，调色板识别变差
- 图纸颜色和界面颜色互相干扰

## 5. 主题系统设计原则

### 5.1 先抽象语义，再做主题皮肤

不要直接写：

- `--beige-1`
- `--beige-2`
- `--orange-1`

优先写语义变量，例如：

- `--theme-bg-page`
- `--theme-surface-panel`
- `--theme-surface-popup`
- `--theme-border-soft`
- `--theme-accent-primary`

### 5.2 主题变量只管界面，不管拼豆颜色数据

拼豆颜色库仍保留现有业务定义。  
主题系统不能改：

- 调色板数据源
- 图纸颜色映射逻辑
- 导出图纸颜色结果

### 5.3 先支持一套默认主题稳定落地

第一阶段只要做到：

- 当前默认主题完整走变量
- 页面整体统一

就已经达标。  
真正的“多主题切换”放在第二阶段做。

### 5.4 主题切换必须零侵入组件逻辑

理想结构应是：

- 根节点切 `data-theme`
- 组件只吃语义 class 和 CSS 变量
- React 组件不写主题判断分支

## 6. 建议的主题变量结构

建议将现有 token 拆成 5 层。

### 6.1 基础原子色

保留现有基础原子色，作为主题底层原料。

例如：

- `--color-ink-strong`
- `--color-ink-soft`
- `--color-bg-page`
- `--color-accent-primary`
- `--color-success`

这层不直接给组件用，更多用于主题映射。

### 6.2 语义主题色

新增统一主题语义变量，作为组件直接消费层。

建议新增：

- `--theme-bg-page`
- `--theme-bg-page-elevated`
- `--theme-surface-panel`
- `--theme-surface-panel-muted`
- `--theme-surface-panel-strong`
- `--theme-surface-popup`
- `--theme-surface-stage`
- `--theme-surface-input`
- `--theme-surface-input-hover`
- `--theme-surface-button-secondary`
- `--theme-surface-button-secondary-hover`
- `--theme-surface-button-ghost`

- `--theme-text-primary`
- `--theme-text-secondary`
- `--theme-text-muted`
- `--theme-text-on-accent`

- `--theme-border-soft`
- `--theme-border-default`
- `--theme-border-strong`
- `--theme-border-focus`

- `--theme-accent-primary`
- `--theme-accent-primary-hover`
- `--theme-accent-primary-active`
- `--theme-accent-secondary`
- `--theme-accent-highlight`

- `--theme-success`
- `--theme-warning`
- `--theme-error`
- `--theme-info`

### 6.3 渐变变量

当前大量渐变直接写在 `globals.css`，建议抽成主题变量。

例如：

- `--theme-gradient-page`
- `--theme-gradient-panel`
- `--theme-gradient-stage`
- `--theme-gradient-upload`
- `--theme-gradient-preview`
- `--theme-gradient-brand-card`

### 6.4 阴影变量

当前已有基础阴影，但建议拆语义层。

例如：

- `--theme-shadow-soft`
- `--theme-shadow-panel`
- `--theme-shadow-floating`
- `--theme-shadow-brand`
- `--theme-shadow-focus`

### 6.5 特殊控件变量

针对工具站关键区域，单独抽局部主题变量。

例如：

- `--theme-topbar-bg`
- `--theme-topbar-border`
- `--theme-stage-frame-bg`
- `--theme-stage-canvas-bg`
- `--theme-help-overlay-bg`
- `--theme-scrollbar-thumb`
- `--theme-scrollbar-track`
- `--theme-selection-active`
- `--theme-chip-active-bg`
- `--theme-chip-active-border`

## 7. 主题组织方式

推荐采用：

```css
:root {
  /* 默认主题变量 */
}

:root[data-theme="atelier"] {
  /* 米白工坊 */
}

:root[data-theme="paper"] {
  /* 冷白纸面 */
}

:root[data-theme="night"] {
  /* 深色夜间 */
}
```

建议第一阶段只保留：

- `atelier` 作为默认主题

第二阶段再增加：

- `paper`
- `night`

## 8. 第一阶段改造范围

第一阶段只做“默认主题彻底语义化”。

优先改这些区域：

### 8.1 全局层

- `body`
- 页面背景
- 全局滚动条
- 全局文字颜色
- 焦点态

### 8.2 容器层

- `.app-shell`
- `.topbar`
- `.panel-card`
- `.modal-sheet`
- `.notification-card`

### 8.3 表单与控件层

- `.button`
- `.field__input`
- `.switch-toggle`
- `.palette-swatch`
- `.replace-card`

### 8.4 编辑器关键层

- `.stage-toolbar`
- `.stage-frame`
- `.stage-frame__canvas`
- `.editor-sidebar`
- `.help-center`

### 8.5 首页层

- `.home-entry__hero`
- `.entry-flow`
- 品牌图标承载底板

## 9. 第二阶段改造范围

在第一阶段稳定后，再做多主题支持。

建议按下面顺序：

1. 新增第二套亮色主题
- 例如更偏冷白、更像设计工具的“纸面工作台”

2. 新增夜间主题
- 但不要直接照搬泛滥的蓝紫暗色
- 需要保留拼豆调色板辨识度

3. 新增主题切换入口
- 放在项目信息弹窗或设置区
- 不建议先放首页

## 10. 不建议主题化的区域

以下区域应谨慎处理，不能跟着页面主题大幅变色：

### 10.1 拼豆颜色库本身

不能因为主题切换去改变：

- 颜色库色值
- 图纸颜色结果
- 颜色统计真实色块

### 10.2 图纸画布内容

画布里的拼豆结果是业务输出，不是 UI 装饰。  
主题只能影响：

- 舞台外壳
- 网格辅助线
- 选区辅助显示

不能影响：

- bead grid 的真实颜色

### 10.3 导出文件

导出 PNG / JSON / 颜色清单不应该跟主题绑定。  
主题只影响编辑器界面，不影响结果文件。

## 11. 推荐主题方向

建议未来先做 3 套，够用，不要一上来太多。

### 11.1 米白工坊

定位：

- 当前默认主题升级版
- 更像手作工具、纸面、木质工作台

关键词：

- 温和
- 明亮
- 手作感
- 不脏

### 11.2 冷白工作台

定位：

- 更像专业工具软件
- 适合追求清爽和现代感的用户

关键词：

- 克制
- 更轻
- 更中性
- 更像设计软件

### 11.3 夜间工作台

定位：

- 长时间编辑使用
- 但必须保证调色板识别度

关键词：

- 深灰
- 低饱和
- 不发紫
- 不赛博

## 12. 实施步骤建议

### 第一步：梳理硬编码

在 `globals.css` 中把以下内容做一次分类登记：

- 直接十六进制
- 直接 rgba
- 直接 gradient
- 直接 shadow

输出一份“待抽离清单”。

### 第二步：补主题语义 token

在 `tokens.css` 中新增主题语义变量层。  
先不要改视觉，只把名字和映射补齐。

### 第三步：把 globals.css 改为只消费语义 token

要求：

- 核心容器不能再直接写死颜色
- 状态色尽量引用主题变量
- 渐变和阴影也走变量

### 第四步：验收默认主题

先不做切换，只看默认主题是否统一：

- 首页
- 顶栏
- 左右侧栏
- 中间舞台
- 弹窗
- 通知

### 第五步：新增第二套主题

以 `data-theme="paper"` 为试点。  
如果第二套主题能基本无痛落地，说明主题系统结构成立。

### 第六步：增加切换功能

最后才做：

- 本地持久化
- 切换入口
- 默认跟随系统或默认固定主题

## 13. 验收标准

主题系统改造完成后，至少满足以下标准：

### 结构标准

- `globals.css` 中显著减少硬编码颜色
- 核心容器改为消费语义变量
- 新增主题时不需要改 React 组件结构

### 视觉标准

- 首页和编辑器属于同一视觉体系
- 左右栏、弹窗、通知、按钮视觉语言统一
- 不再出现“局部单改还行，整体越改越乱”的情况

### 功能标准

- 主题切换不影响拼豆颜色数据
- 不影响导出结果
- 不影响画布渲染逻辑

## 14. 风险与注意事项

### 14.1 一次性全量替换风险较高

不建议一次性把 `globals.css` 所有硬编码全改掉。  
应该按区域推进：

- 先全局
- 再容器
- 再控件
- 再右栏细节

### 14.2 当前页面“丑”的原因不只是颜色

还有：

- 边框太多
- 卡片层级太碎
- 间距密度不统一
- 某些控件像表单后台，不像工具站

所以主题系统只能解决“颜色体系混乱”，不能单独解决所有视觉问题。

### 14.3 主题系统不等于 UI 重设计

主题系统是基础设施。  
UI 变好看，还需要后续继续处理：

- 组件层级
- 信息密度
- 控件尺寸
- 图形化表达

## 15. 建议结论

建议立刻推进主题系统改造，但顺序必须是：

1. 先做语义 token 层
2. 先收一版默认主题
3. 再做第二套主题验证结构
4. 最后再加主题切换功能

不要反过来先做主题切换按钮。  
否则只是把混乱的样式换着切，不能真正解决问题。
